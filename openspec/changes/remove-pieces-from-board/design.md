## Context

The puzzle app supports manual piece placement via drag-and-drop from the PiecesList sidebar onto the board. Once placed, the only way to remove a piece is to double-click its entry in the PiecesList (shown at 50% opacity). There is no direct interaction with placed pieces on the board, no visual hint that placed pieces are removable, and no bulk-clear option.

State management is centralised in `App.js`:
- `solution` array holds all placements as `{ pieceId, cells }` objects
- `pieces[].used` boolean tracks piece availability
- `handlePieceRestore(pieceId)` already implements the removal logic (filters `solution`, resets `used` flag)

The board (`Board.js`) currently renders placed pieces as coloured cells but has no click handlers for them.

## Goals / Non-Goals

**Goals:**
- Allow users to click a placed piece on the board to remove it
- Provide clear visual feedback (hover highlight, cursor) that placed pieces are interactive
- Add a "Clear Board" button to remove all manually-placed pieces at once
- Keep the existing double-click restore in PiecesList as an alternative path

**Non-Goals:**
- Undo/redo history for placements
- Confirmation dialogs before removing pieces
- Backend changes or API modifications
- Modifying the automated solver flow — "Clear Board" only affects manual placements
- Touch/mobile gesture support (the app currently targets desktop only)

## Decisions

### 1. Click-to-remove on board cells
**Decision**: A single click on any cell belonging to a placed piece removes that entire piece.

**Rationale**: Single click is the most natural and discoverable interaction. Double-click (as used in PiecesList) is less discoverable. Since board cells don't currently have a click handler, there's no conflict.

**Alternative considered**: Right-click context menu — rejected as over-engineered for a single action and less discoverable.

### 2. Identify piece from clicked cell via existing `solution` array
**Decision**: On cell click, look up which piece occupies that cell by searching the `solution` array, then call the existing `handlePieceRestore(pieceId)` function.

**Rationale**: Reuses the existing removal logic. No new state management needed. The `solution` array is already the source of truth.

### 3. Hover affordance for removable cells
**Decision**: Placed piece cells get a CSS hover effect (brightness change + pointer cursor) to signal interactivity.

**Rationale**: Minimal visual change that clearly signals "this is clickable" without altering the existing colour scheme.

### 4. "Clear Board" button placement
**Decision**: Place the "Clear Board" button next to the existing "Solve" button in the control area of the UI. Only enabled when there are manual placements (i.e., `solution` is non-empty and the solution was not auto-generated by the solver).

**Rationale**: Co-locating with "Solve" keeps puzzle actions grouped. Disabling when empty prevents confusion.

### 5. Solver vs. manual solution distinction
**Decision**: "Clear Board" clears all placements and resets all pieces. It applies to both solver-generated and manually-placed solutions — effectively a full board reset.

**Rationale**: Simplest mental model. Users expect "Clear Board" to mean "start fresh." Differentiating between solver and manual placements adds complexity with little benefit.

## Risks / Trade-offs

- **[Accidental removal]** → Single click makes it easy to accidentally remove a piece. Mitigated by the simplicity of re-placing a piece via drag-and-drop. No confirmation dialog to keep the flow fast.
- **[Solution array scan on every click]** → Searching `solution` for the clicked cell is O(total placed cells). With a maximum of 41 cells across 9 pieces, this is negligible.
- **[Click vs. drag conflict]** → A click handler on board cells could interfere with drag-and-drop placement. Mitigated by only attaching the click-to-remove handler to cells that are already filled (have a piece), not to empty fillable cells.
